0 Executive Summary

SureRide is a production-ready, AI-powered conversational agent designed to address one of Kenya's most acute road-safety challenges: drunk driving. According to the National Transport and Safety Authority (NTSA), alcohol-related crashes account for over 30% of annual road fatalities in Kenya. The core friction is not a lack of awareness — it is the difficulty of booking a safe ride home late at night, when impairment degrades decision-making and digital fluency.

SureRide eliminates this friction by deploying a stateful LangGraph conversational agent backed by a Retrieval-Augmented Generation (RAG) knowledge base and a live PesaPal payment gateway. The system guides a user from greeting to confirmed booking in under 60 seconds, via a familiar WhatsApp-style chat interface, without requiring any app download or account creation.

1 Problem Framing

1.1 The Problem — Drunk Driving in Kenya

Drunk driving is a preventable public health crisis. The NTSA attributes more than 30% of Kenya's annual road accident fatalities to alcohol-impaired driving — a figure that spikes significantly on Friday and Saturday nights. The problem is not purely behavioural: impaired individuals intend to get home safely but lack a fast, low-friction way to do so at 2 AM outside a Nairobi nightspot.

Existing ride-hailing applications (Uber, Bolt, Little) require account registration, credit card setup, and reliable mobile data — barriers that are disproportionately high for impaired users. The friction of multi-step app navigation in a degraded cognitive state is precisely the gap SureRide is designed to close.

1.2 Why an AI Agent is the Right Solution

AI Justification: A conversational agent mimics WhatsApp — the most familiar digital interface for Kenyan mobile users. It guides the user through a booking with natural-language prompts, requiring only one answer at a time, dramatically lowering cognitive load compared to a traditional form-based app.

Three criteria confirm AI agents as the appropriate solution type:

  • Impact: Directly addresses a high-severity organizational and societal friction point with a quantifiable target outcome (reduce drunk-driving incidents).
  • Justification: The use of a stateful LangGraph agent enables multi-turn conversation with memory, something rule-based IVR or simple chatbots cannot achieve reliably.
  • Feasibility: The project scope (19 pilot zones, single-session booking, simulated driver dispatch) is calibrated for a 12-week design cycle with one developer.

2 System Architecture & Technical Execution

2.1 Architecture Overview

SureRide is built on four decoupled layers, each independently testable:

Layer 1 — Conversation State Machine (langgraph_agent/)
A LangGraph StateGraph with a single dispatcher node that reads current_step from BookingState and delegates to the correct handler. The dispatcher pattern prevents loop traps, makes transitions fully deterministic, and isolates user input from the LLM system prompt.
Layer 2 — RAG Knowledge Base (rag/)
ingest.py chunks the sureride_faq.txt (chunk size 400, overlap 60) and embeds it using Google Gemini embedding-001 into a local ChromaDB vector store. retriever.py returns a LangChain retriever (k=3) used by the rag_tool_node for mid-booking FAQ interception.
Layer 3 — Business Logic Utilities (utils/)
zones.py — 19 zone centroids with alias lookup, substring match, and difflib fuzzy matching.
pricing.py — FareBreakdown dataclass with Haversine × 1.3 road distance and surge logic.
pesapal.py — PesaPal v3 OAuth2 flow: token → IPN registration → order submission → status polling.
Layer 4 — Streamlit UI (ui/app.py)
WhatsApp-style chat rendered inside an iframe component to isolate CSS. Sidebar navigation exposes an Operations Dashboard with KPI metrics, zone demand bar chart, ethics KPIs, and a transaction feed — providing the required practical dashboard displaying core metrics.

2.2 Booking Conversation Flow

greet → collect_name → collect_pickup → validate_pickup → collect_destination → validate_destination → show_fare → confirm_fare → await_payment → done

At any collect_* step, if the user's message begins with a question word or ends with "?", the dispatcher intercepts it and routes to rag_tool_node, returning an FAQ answer before resuming the booking at the same step.

3 Prompt Engineering Strategy

3.1 RAG FAQ Grounding Prompt

The primary LLM call in the system is the FAQ synthesis prompt used by rag_tool_node:

You are the SureRide AI assistant. Use ONLY the context below to answer
the customer's question concisely and in a friendly, conversational tone.
Do not make up information. Keep the answer under 4 sentences.

Context:
{retrieved_chunks}

Customer question: {user_query}

Answer:

Design principles applied:

  • Role constraint — sets persona, tone, and domain boundary.
  • Source grounding — "Use ONLY the context below" prevents hallucinated pricing, ETAs, or phone numbers.
  • Conciseness constraint — "under 4 sentences" keeps responses within a WhatsApp bubble.
  • Zero-shot instruction — "friendly, conversational tone" ensures natural output without few-shot examples.

3.2 Deterministic Node Strategy (Anti-Prompt-Injection)

Security design decision: For all structured booking steps (name, pickup, destination, confirmation), user input is processed by deterministic Python heuristics — never by an LLM. This renders prompt injection attempts completely ineffective because user messages never reach the LLM system prompt in those steps.

Example from collect_name_node:

last = state["messages"][-1].content.strip()
name = last.split()[0].capitalize()   # first word only, capitalised

4.4 Explainability

The SureRide agent is fully deterministic and rule-based at the conversation-flow level. Every decision traces back to a single field in BookingState.current_step. An auditor can inspect the full state at any point to understand exactly what the agent recorded and why. There is no black-box decision process in the booking pipeline.

For RAG answers, the retrieved source chunks are traceable to specific lines in sureride_faq.txt, making every FAQ answer independently verifiable against the company knowledge base.

6 Reflection & Architectural Limitations

6.1 What Worked Well

  • Dispatcher pattern in LangGraph — the single-node, step-routing architecture proved highly robust. It eliminated loop traps that appeared in early multi-node experiments and made debugging trivial: every failure traced to a single handler function.
  • RAG grounding — restricting the LLM to "ONLY the context below" completely eliminated hallucinated prices and policies in FAQ answers. Every answer was traceable to a specific line in sureride_faq.txt.
  • PesaPal integration — implementing a real payment gateway (with mock fallback) rather than a simulated one gave the demo genuine credibility. The mock mode transparently labels itself, satisfying both demo practicality and ethical honesty requirements.
  • WhatsApp-style UI — rendering chat bubbles inside an iframe isolated the CSS from Streamlit's stylesheet, producing a visually authentic interface that resonated strongly with the target user context.

6.2 Technical Challenges Encountered

  • Streamlit iframe rendering — Streamlit's layout engine strips custom CSS from st.markdown(). Solution: use components.html() to render an isolated iframe with full CSS control, at the cost of added height-estimation complexity.
  • PesaPal webhook limitations — the local Streamlit server has no public callback URL, making PesaPal's IPN webhook system unusable in development. Mitigation: implemented a manual polling button that queries the PesaPal status API directly using OAuth2 token validation.
  • LangGraph state accumulation — the add_messages reducer appends messages rather than replacing them, causing early versions to re-trigger greet after every turn. Fixed by adding the not messages or messages[-1].type != "human" guard in the dispatcher.

6.3 Current Limitations

LimitationImpact
No real driver dispatchBooking confirmation is a simulation; no actual driver is assigned
Single-session memoryReturning users start fresh — no booking history or preferences
Heuristic name parsingFirst-word split fails for compound names or greeting responses ("Hi, I'm John")
Haversine distance onlyRoad distance = haversine × 1.3 (fixed tortuosity factor); no real routing data
English onlyNo Swahili support — excludes a significant portion of Kenyan users
19 zones onlyPeri-urban areas (Thika, Ruiru, Athi River) are excluded from pilot

6.4 Recommended Next Steps

  1. Integrate Google Maps Directions API for exact road distance (replace Haversine × 1.3).
  2. Add Swahili language support using a multilingual LLM (Gemini 1.5 Pro natively supports Swahili).
  3. Replace heuristic name extraction with a named-entity recognition (NER) model.
  4. Implement KDPA-compliant PostgreSQL persistence for booking history with row-level encryption.
  5. Conduct a formal algorithmic bias audit before any production driver-matching ML model.
  6. Commission an independent KDPA Data Protection Impact Assessment (DPIA).
  7. Deploy on a public cloud endpoint (Cloud Run / Railway) with a real PesaPal IPN webhook URL.

6.5 Lessons Learned

The most valuable lesson from this capstone was that responsible AI design is not an afterthought. The choice to use a hard-coded LangGraph dispatcher — rather than a free-form LLM — was driven equally by ethical reasoning (preventing hallucinated bookings) and engineering preference. The decision to delegate payment to PesaPal's hosted pages was the correct KDPA compliance call, even though it added integration complexity. This principle — that safety constraints and ethical guardrails often produce better engineering outcomes — is one to carry into all future AI system design.
